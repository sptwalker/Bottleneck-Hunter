## 一、系统概况

### 1.1 工程规模

| 维度 | 数据 |
|------|------|

| 前端 JS/CSS/HTML | 34 文件，22,393 行 |

| 核心模块 | chain(6,155行) / watchlist(7,957行) / web(4,768行) / auth(618行) |

### 1.2 架构总览

```
产业链拆解(Phase 1-4)  ──→  观察池(Watchlist)  ──→  四层决策引擎(L1-L4)
      │                          │                        │
      ├─ 瓶颈识别                ├─ 5 条数据管道           ├─ 多 LLM 投委会
      ├─ 供应商筛选              ├─ 双市场(美股/A股)       ├─ 模拟交易执行
      └─ 交叉验证               └─ SSE 实时刷新          └─ 闭环反馈系统
```

### 1.3 核心设计理念

系统采用 **"LLM 做判断，规则做感知"** 的架构——规则系统负责收集和结构化数据，LLM 负责所有投资判断。四层决策（宏观策略→组合配置→战术计划→账户执行）加上投委会交叉验证，设计理念在同类系统中属于**领先水平**。

## 四、分项深度评估

### 4.1 产业链瓶颈分析系统

#### 优势
- **独创的方法论**：从产业链瓶颈切入选股，避开了拥挤的因子投资和动量策略
- **结构化拆解**：3+ 层递归拆解，支持预设模板加速，去重机制考虑周全
- **五维评分模型**：稀缺性/不可替代性/供需缺口/定价权/技术壁垒覆盖了瓶颈的核心维度
- **交叉验证**：Phase 4 的多模型验证降低了单一 LLM 的幻觉风险

#### 🔴 Critical 问题

**C-1：「瓶颈环节的供应商 ≠ 好的投资标的」**

这是整个系统最根本的逻辑漏洞。系统假设：识别到瓶颈环节 → 该环节的上市公司就值得投资。但现实中：

| 场景 | 瓶颈度 | 投资价值 | 原因 |
|------|--------|---------|------|
| 磷化铟衬底（国内唯一供应商） | 9/10 | ❌ 低 | 市场规模 <10 亿，客户仅 3 家 |
| 光刻胶（国产替代） | 8/10 | ⚠️ 高风险 | 只能做低端 DUV，定价权弱 |
| HBM 高带宽内存（SK 海力士） | 8/10 | ✅ 高 | 市场规模 100+ 亿，垄断定价 |

**缺失的过滤维度**：市场规模（<10 亿不投）、下游客户数（<5 个不投）、毛利率（<25% 不投）、收入规模。

**C-2：LLM 拆解的不可重复性**

每次调用 `decomposer.py` 对同一产品拆解，结果都可能不同。没有版本管理、基准节点库或结果缓存。这导致：
- 同一产业链多次拆解得到不同的瓶颈排名
- 无法进行参数敏感性分析
- 无法追踪"为什么得到这个产业链配置"

**C-3：瓶颈评分完全依赖 LLM 主观打分**

5 个维度的 0-10 分全部由 LLM 凭"世界知识"评判，没有数据支撑和证据链。不同模型的评分差异可能达 2-3 分。

#### 🟠 High 问题

- **权重设计缺乏行业验证**：稀缺性和不可替代性各占 0.25，但两者高度相关（ρ≈0.8），导致稀缺维度过度放大
- **深度 3 层可能不够**：GPU 产业链的真正瓶颈（如磷化铟晶体生长设备）在第 4-5 层，被深度限制截断
- **财务数据混合权重双重强化**：`_blend()` 给数据驱动 0.7 权重后，加权平均又给财务 1.5 倍系数

### 4.2 交易决策引擎

#### 优势
- **四层递进架构**：L1 宏观→L2 组合→L3 战术→L4 执行，每层职责清晰、更新频率合理
- **增量更新策略**：L1 日常检查而非完全重建，降低成本和决策漂移
- **投委会机制**：4 个角色（风控官/成长/价值/逆向）从不同 LLM 提供商独立评审
- **共识规则清晰**：60% 通过率、50% 共识修改、少数派意见记录

#### 🔴 Critical 问题

**C-4：决策闭环完全断裂**

设计文档描述了完整的闭环：执行→反馈→复盘→经验卡片→策略优化。但从数据库实际数据看：

```
设计中的闭环:  L4执行 → 用户确认 → 模拟成交 → 自动复盘 → 经验卡片 → L2策略优化
实际运行:      L4执行 → (断裂) → 0 条模拟交易 → 0 条复盘 → 0 张经验卡片
```

`trade_executor.py` 和 `trade_reviewer.py` 的关键逻辑缺失或未运行。

**C-5：投委会圆桌讨论缺失实现**

设计文档 §4.5 定义了 2v2 分歧时的圆桌讨论机制，但代码中完全跳过此步骤，直接进入共识汇总。投委会变成了 4 次独立投票，失去了辩论和共识凝聚的价值。

**C-6：L4 执行计划无约束硬验证**

5 个硬性约束（单股 20%、单板块 40%、现金 15%、单笔 $1K、日交易 30%）写在 prompt 中让 LLM 遵守，但**没有代码层的验证**。LLM 可能"遗忘"约束，生成违规计划直接写入数据库。

#### 🟠 High 问题

- **L1 宏观数据输入为空**：`_collect_market_context()` 的 `macro` 字段返回空字典，L1 在没有 VIX/美债/联储数据的情况下判断市场阶段
- **L1→L2 数据流不清晰**：L1 的板块建议如何量化为 L2 的权重分配，没有映射规则
- **催化剂结果判定未实现**：催化剂日期过后的 realized/failed/neutral 判定逻辑缺失
- **用户偏好学习机制未实现**：L4 的 `{user_preferences}` 替换始终为空字典

### 4.3 数据管道系统

#### 优势
- **双市场覆盖**：美股（yfinance + SEC EDGAR）和 A 股（akshare）独立管道
- **5 条管道并行**：价格/新闻/SEC/期权/公告，异步 + Semaphore 并发控制
- **调度系统成熟**：APScheduler 14 个定时任务，盘前盘后+日终+周度分层刷新
- **部分失败隔离**：partial 状态标记，单个标的失败不影响其他
- **重试机制**：指数退避 + 可配置重试次数

#### 🔴 Critical 问题

**C-7：SQLite 多线程安全**

`check_same_thread=False` 在 FastAPI 异步环境下可能导致数据损坏。虽然当前通过每次调用新建连接缓解，但高并发场景仍有风险。

#### 🟠 High 问题

| 问题 | 影响 | 当前状态 |
|------|------|---------|
| **无盘中实时数据** | 只能日频决策，无法日内交易 | 仅日线数据 |
| **Form 4 内幕交易解析完全不可用** | shares=0, price=None, name=Unknown | 49 行全是占位数据 |
| **美股夏令时未处理** | 11 月-3 月定时任务时间错 1 小时 | 硬编码 UTC 偏移 |
| **market_snapshots 无 market 列** | H 股+A 股同 ticker 数据覆盖 | 缺字段 |
| **机构持仓、分析师评级完全缺失** | 两个最重要的基本面维度为零 | 无数据源 |
| **A 股数据严重弱于美股** | 无期权、无基本面、无机构持仓 | 仅价格+新闻+公告 |

### 4.4 风险控制与回测

#### 优势
- **约束定义完整**：仓位、板块、现金、交易规模约束已定义
- **投委会风控官角色**：专门的风险控制视角，涵盖压力测试、流动性评估
- **LLM 成本预算系统**：BudgetTracker 三级降级（FULL→REDUCED→MINIMAL）
- **模拟账户基础设施**：sim_account/sim_positions/sim_trades 表结构完备

#### 🔴 Critical 问题

**C-8：无系统化风控框架**

系统的风险控制完全停留在"定义"层面，缺乏执行层面的保障：

```
需要的:                          实际的:
├─ 仓位约束硬编码执行             ├─ 写在 prompt 里让 LLM 遵守
├─ VaR/CVaR 每日计算             ├─ 不存在
├─ Portfolio Beta 监控            ├─ 不存在
├─ 关联性风险矩阵                ├─ 不存在
├─ 自动熔断（跌幅>15%）          ├─ 不存在
├─ 流动性评估                    ├─ 不存在
└─ 尾部风险对冲建议              └─ 不存在
```

**C-9：回测能力几乎为零**

| 回测维度 | 现状 |
|---------|------|
| 历史信号回放 | ❌ 无法重放历史时点的分析结果 |
| 参数敏感性测试 | ❌ 无法改变权重后重算历史分数 |
| Sharpe Ratio | ❌ 不计算 |
| Sortino Ratio | ❌ 不计算 |
| Information Ratio | ❌ 不计算 |
| 最大回撤分析 | ❌ 不计算 |
| 基准对比 | ❌ 无基准收益率 |
| 策略胜率统计 | ❌ sim_trades 为空 |

没有回测能力意味着**无法验证系统的投资建议是否有效**。这是量化系统最基本的要求。

#### 🟠 High 问题

- **仓位管理无科学算法**：无凯利公式、无风险均衡、无波动率缩放
- **组合级风险度量全部缺失**：无 Portfolio Beta、Correlation Matrix、Concentration Index
- **尾部风险无处理**：无黑天鹅预案、无自动减仓机制
- **关联性风险盲区**：同一产业链上下游持仓的集中度风险无法识别

## 六、量化分析师视角的核心痛点

### 痛点 1：无法回答"这个系统赚钱吗"

没有回测数据，就无法计算 Sharpe Ratio、年化收益、最大回撤。一个量化分析师拿到这个系统的第一个问题就是："给我看历史表现"——而系统目前无法回答。

### 痛点 2：风控只存在于纸面上

约束写在 prompt 里而非代码里，等于没有约束。在极端市场条件下（如 Flash Crash），LLM 可能生成违规操作建议，系统无法拦截。

### 痛点 3：信号→执行→反馈的链路中断

L1-L4 生成了完整的投资建议，投委会也做了交叉验证，但到"实际执行"这一步戛然而止。没有模拟成交、没有复盘、没有经验积累——决策系统无法从自己的错误中学习。

### 痛点 4：数据基础不够扎实

机构持仓、分析师评级这两个最重要的基本面维度完全缺失。内幕交易数据全是占位符。A 股比美股弱一个量级。在这样的数据基础上做投资决策，风险极高。

### 痛点 5：产业链逻辑的最后一公里没打通

从"识别瓶颈"到"推荐投资标的"之间缺少关键的可投性过滤。瓶颈度高的环节，其供应商不一定是好投资（市场规模太小、定价权弱、客户集中）。

## 八、总体结论

### 8.1 系统定位

BottleneckHunter 目前是一个**设计先进但实现半成品的投资辅助工具**。

- **作为产业链研究工具**：✅ 可用。拆解引擎和瓶颈识别在辅助研究方面有独到价值。
- **作为日常监控工具**：⚠️ 部分可用。观察池 + 数据管道 + 新闻聚合提供了基本的监控能力。
- **作为交易决策系统**：❌ 不可用。闭环断裂 + 无回测 + 风控松散，不应用于实际投资决策。
- **作为量化策略平台**：❌ 严重不足。缺乏回测、风险度量、组合优化的基本能力。

### 8.2 核心优势（应保持）

1. **独创的产业链瓶颈选股方法论** — 差异化的 Alpha 来源
2. **四层递进决策架构** — 学术级的系统设计
3. **多 LLM 投委会交叉验证** — 有效降低单模型偏差
4. **完整的闭环反馈设计** — 虽未实现，但框架正确
5. **多用户系统** — 工程层面的成熟度

### 8.3 最紧迫的改进方向

1. **先把闭环跑通**（Phase A+B 的核心任务），让系统能自我学习
2. **建立回测能力**，用数据说话而非"感觉"
3. **硬编码风控约束**，不能指望 LLM 永远遵守规则
4. **补全关键数据维度**，特别是机构持仓和分析师评级

### 8.4 量化分析师的一句话评价

> **"这是一套拥有教科书级架构设计的系统，但在从设计到实现的最后一公里断裂了。补齐闭环和风控后，它有潜力成为独特而有效的产业链量化投资工具。"**
