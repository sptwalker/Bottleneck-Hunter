# VIP 私人财务 AI 顾问团队 —— 地基工程规划

> 状态：规划（Planning）。本文件是路线图/里程碑；详细技术方案见 `docs/VIP_ADVISOR_TECH_SPEC.md`。
> 目标：VIP 用户导入券商月结单/交易 PDF → 系统解析归类、学习持仓结构与交易习惯、为衍生品建模 →
> 用决策中心同源的多层 AI 团队（宏观→配置→操作）定期出投资分析报告与持仓建议、不定期推荐 →
> 提供实时咨询窗口随时答疑。

## 0. 核心原则

- **能复用决策中心的绝不重建**。真正要新建的，是"从用户真实对账单到可被 AI 推理的数据"这条链，
  以及财务级隐私安全。
- **窄而深先行**：先用一个垂直切片验证最大风险（PDF 抽取准确率 + 数据模型能否喂决策），
  再横向扩券商/品类、纵向加语义检索/聊天。
- **财务数据不能偷懒**：准确性、隐私加密、合规免责三条是硬底线，先于任何功能就位。

## 1. 复用 vs 新建

**直接复用（已成熟）**
- L1–L4 决策引擎 + 投委会（`watchlist/decision_engine.py`、`committee.py`）
- 多用户/市场隔离（`store.for_user(sub).for_market(market)`）
- LLM 工厂：多 provider + 自动 fallback + 角色矩阵 + 全局限速闸门（`llm_clients/factory.py`、`data_provider/yf_gate.py`）
- 数据源层（标的行情，供衍生品定价用）（`data_provider/`）
- SSE 流式（`web/streaming/`、`static/js/sse.js`）
- 预算护栏 `BudgetTracker`、备份 `scripts/backup.py`、可观测 Loki
- 经验卡片知识底座（`store_research.py`：experience_cards / auto_reviews / layer_performance / theses）

**必须新建（本功能独特地基）**
1. 对账单/PDF 摄取与归一化管道（最难最脆）
2. 金融规范数据模型（持仓/交易/工具，尤其衍生品）
3. 金融隐私安全层（真实 PII，摄取前必须就位）
4. 用户画像蒸馏 + 衰减 + 语义检索（把海量原始交易变成可注入的紧凑记忆）
5. 真实组合 → 决策引擎适配器 + 咨询会话记忆

## 2. 依赖脊柱

```
安全层(P0) ─┐
            ├─→ 摄取管道(P1) ─→ 规范模型(P2) ─→ 蒸馏画像(P3) ─→ 语义检索(P4) ─→ 决策适配(P5) ─→ 报告/推荐 + 咨询聊天(P6)
衍生品建模(P2b)┘
```
安全层与数据模型是最底层——**在导入任何真实财务数据之前必须先有，不可事后补**。

## 3. 地基分阶段（按依赖 + ROI 排序）

### P0 · 安全与合规底座（最先，不产出功能）
- 对账单原文件 + 解析结果**静态加密**（复用现有"按用户加密 Key"机制，扩展到财务文档）；原始 PDF 处理后可选即焚只留结构化数据。
- 强隔离 + VIP tier 门禁（复用邀请码/tier）+ **用户删除权**（一键抹除全部财务数据）+ 每条建议**审计留痕**（文件→数据→建议可溯源）。
- **合规免责**：所有报告/聊天输出统一挂免责声明、不承诺收益。硬底线。

### P1 · 摄取管道（最高风险，最早验证）
- PDF→结构化：券商格式千差万别 → 用 **LLM 抽取（文本/视觉）+ 严格 schema 校验 + 人在环校正**，不硬写每家解析器。先跑通 1–2 家券商的股票对账单。
- 管道：上传→解析→校验→**去重**（月结单跨月重叠）→**对账**（持仓 vs 交易推导持仓要对得上，对不上标红让用户确认）→落库，**全程 provenance**（溯源到文件+页）。
- ⚠️ 成败点：先用窄切片证明抽取准确率，别一上来全券商全品类。

### P2 · 规范数据模型 + 衍生品建模
- 规范模型：`instruments`（工具主数据）/`positions`（持仓快照）/`transactions`（流水），统一多币种多品类。
- 衍生品数据模型（用户明确要求）：期权（标的/行权价/到期/乘数/方向）→ **Black-Scholes 定价 + Greeks**；结构化产品 → **条款化现金流模型**。标的行情复用 `data_provider`。

### P3 · 用户画像蒸馏 + 衰减（ROI #1）
- 把海量原始交易**蒸馏**成始终最新的紧凑"用户画像"（持仓结构、交易风格、风险承受、盈亏习惯），每次 LLM 调用便宜注入，而非塞几个月流水。
- 定时**衰减/归并**陈旧记录防噪声；复用经验卡片 confidence/applied 思路。
- 是决策适配与聊天的共同前置，也对现有决策中心立即有用。

### P4 · 语义检索层（ROI #2）
- 现有表**加一层本地 embedding 索引**（sqlite-vec / FAISS，纯附加不动主结构），索引：用户交易历史、历次报告、聊天记录、全局经验卡片。
- 让报告/聊天能召回**类比情境**（相似持仓/市况下的历史教训与建议），非只精确匹配。唯一真需要"向量"的地方。

### P5 · 决策适配 + 报告/推荐（大量复用）
- 适配器：真实组合数据 → 喂现有 L1–L4 + 投委会（L1 宏观共享 / L2 基于真实持仓的配置 / L3 逐仓战术 / L4 具体操作：调仓/对冲/止盈）。工程量在接口适配。
- 定期报告 + 事件驱动推荐：复用 `chain/report.py` 与调度器。

### P6 · 实时咨询聊天
- RAG 聊天：接语义检索（组合+历史+报告）+ 实时行情工具 + 知识库；**会话记忆**（新建 chat_sessions 表）；复用 SSE 流式；挂合规护栏。

## 4. 其他必要能力（横切，勿漏）
- **对账/纠错闭环**：解析必然有错，需用户确认/修正界面，错误反哺解析。
- **成本护栏**：VIP 用量重，复用 `BudgetTracker` 做每-VIP 预算。
- **可观测**：Loki 已有，摄取/解析失败要能追。

## 5. 首个里程碑（窄而深垂直切片）
P0 安全 + P1 摄取（**仅 1 家券商股票月结单**）+ P2 规范模型（**先不含衍生品**）+ P3 基础画像 + 复用决策引擎 → 出**一份持仓分析报告**。跑通即验证最大风险。之后横向加券商/衍生品，纵向加语义检索/聊天。

## 6. 阶段与里程碑对照

| 里程碑 | 覆盖阶段 | 验证的关键假设 |
|---|---|---|
| M1 垂直切片 | P0 + P1(窄) + P2(无衍生品) + P3(基础) + P5(报告) | PDF 抽取准确率 + 数据模型能喂决策 |
| M2 知识增强 | P3(完整) + P4 语义检索 | 历史/类比经验能提升建议质量 |
| M3 衍生品 | P2b 衍生品建模 | 期权/结构化产品可建模并纳入建议 |
| M4 实时咨询 | P6 聊天 | RAG 咨询可用、合规可控 |
| M5 横向扩展 | P1 多券商 + 推荐引擎 | 规模化摄取 + 不定期推荐 |

## 7. P1 摄取可行性验证结论（2026-07，真实样本）

用真实月结单（花旗私人银行 Citibank NA 香港 综合月结单，7 期）验证：
- ✅ **原生文本层**（pdftotext 抽出 ~20 万字符/期，无需 OCR）。
- ✅ **逐只持仓可抽**：个股（GOOGL/NVDA/ASML/ARM/MSFT/META/TSLA/AVGO/MRVL/MU/PLTR/TSM/700 HK 等，代码可映射现有 data_provider）带 数量/成本/市值/未实现盈亏；固收带 ISIN；**期权表**带 Trade/Expiry/Call-Put/Contracts/Premium/Mark-to-Market。
- ✅ **选型被真实数据背书**：`-layout` 下列会错位/空缺，确定性正则会错 → 必须 LLM 抽取 + 严格 schema + 语句内对账（D2 选择正确）。
- ⚠️ **偏差修正**：真实首客是 **HNW/私行综合单**（多币种+固收+期权+结构化产品，非零售纯股票单）。
  - **M1 调整**：仍先只抽 **EQUITIES 子表** 跑通全链（最干净、直喂决策），固收/期权/结构化留 M3。
  - **P2b 衍生品建模提前重视**：首客第一天即持有期权，虽不进 M1，但设计上不可轻。
- 构建期正式引入 **PyMuPDF(fitz)**（比 pdftotext 更好的版面/坐标，喂 LLM 更稳）；本机验证用 pdftotext 已足。
